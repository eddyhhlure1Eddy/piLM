# piLM: Μηχανή Αυτο-Συλλογισμού με Ecpu + Eram

> Αυτοκατασκευή μηχανής συλλογισμού υψηλής απόδοσης. **Ecpu** (πυρήνας υπολογισμού) + **Eram** (πυρήνας μνήμης) συνεργατικός συλλογισμός, Python στρώμα προγραμματισμού + C στρώμα πυρήνα, υποστήριξη Speculative Decoding και Reasoning αλυσίδας.

## 0. Ονοματοθεσία

| Όνομα | Σημασία | Αντιστοιχία |
|---|---|---|
| **piLM** | Όνομα έργου (prototype inference Language Model) | Το σύνολο του έργου |
| **Ecpu** | Πυρήνας υπολογισμού (C kernel + dispatch συνόλων εντολών) | Επανασύλληψη της έννοιας CPU |
| **Eram** | Πυρήνας μνήμης (φόρτωση βαρών/KV + βελτιστοποίηση εύρους ζώνης) | Επανασύλληψη της έννοιας DRAM |
| **Ecpu-Eram συνεργασία** | Αρχιτεκτονική συνεργατικού συλλογισμού υπολογισμού-μνήμης | Βασικό παράδειγμα του έργου |

## 1. Προσανατολισμός Έργου

| Διάσταση | Απόφαση |
|---|---|
| Έννοια "αυτο-συλλογισμού" | **Τριπλός συνδυασμός**: αυτοκατασκευή μηχανής + Self-Speculative Decoding + Reasoning αλυσίδα |
| Μορφή συνεργασίας | **Αμιγής Ecpu συλλογισμός + βελτιστοποίηση εύρους ζώνης Eram πολλαπλών καναλιών** (χωρίς GPU offload) |
| Υλικό ανάπτυξης | **Μικτό/απροσδιόριστο** (x86 διακομιστές AMX/AVX-512, επιφάνεια εργασίας AVX2, ARM NEON/SVE — όλα απαιτούν αφαίρεση) |
| Μοντέλα στόχου | **Υποστήριξη πολλαπλών μεγεθών**, από Qwen3.5-9B → Qwen3-8B-Coder GGUF → Qwen3-1.7B, βαθμιαία επικύρωση από μεγάλο σε μικρό |
| Κύριος στόχος | **Μηχανή υψηλής απόδοσης**, προσανατολισμένη προς το άριστο της απόδοσης αμιγούς CPU |

## 2. Αρχιτεκτονική Γλώσσας και Στρωμάτων

```
┌─────────────────────────────────────────────┐
│  Python στρώμα προγραμματισμού              │
│  - προγραμματισμός αιτημάτων, continuous batching
│  - φόρτωση μοντέλου, χάρτης βαρών, tokenizer │
│  - Speculative / Reasoning άγκιστρα         │
├─────────────────────────────────────────────┤
│  C ABI όριο (ctypes / CFFI / pybind)        │
├─────────────────────────────────────────────┤
│  C στρώμα πυρήνα (Ecpu πυρήνας)             │
│  - ποσοτικοποιημένο GEMM / attention / layernorm
│  - dispatch συνόλων εντολών (AMX/AVX512/AVX2/NEON)
│  - threadpool / NUMA συγγένεια              │
├─────────────────────────────────────────────┤
│  Eram στρώμα αποθήκευσης (ενσωματωμένο C)   │
│  - mmap διαμονή βαρών, ανάγνωση κατ' απαίτηση│
│  - KV cache σε σελίδες, θερμή/κρύα στρωμάτωση
│  - αντίληψη εύρους ζώνης πολλαπλών καναλιών │
└─────────────────────────────────────────────┘
```

**Αρχή κατανομής**: Python για ευέλικτο προγραμματισμό και ενορχήστρωση (εύκολη ανάπτυξη, εύκολη επανάληψη), C για επιχειρήσεις ευαίσθητες στην απόδοση και πρόσβαση μνήμης (ακραίο εύρος ζώνης, μηδενικό overhead). Τα δύο στρώματα επικοινωνούν μέσω λεπτού C ABI, τα θερμά μονοπάτια δεν διασχίζουν γλώσσες.

## 3. Βασικοί Στόχοι Σχεδιασμού

1. **Εύρος ζώνης Eram προτεραιότητα**: Το bottleneck του Ecpu συλλογισμού είναι το εύρος ζώνης Eram, όχι ο υπολογισμός. Ο σχεδιασμός πυρήνα πρέπει να διαβάζει κάθε byte βαρών μόνο μία φορά, προσεγγίζοντας το όριο εύρους ζώνης STREAM.
2. **Πλήρης κάλυψη ποσοτικοποίησης**: BF16 / FP8 / INT8 / INT4 (GGUF Q4_K_M / Q5_K / Q8_0) πλήρης υποστήριξη, βάρη μόνιμα στο Eram.
3. **Στρωματωτός KV Cache**: θερμά token KV σε L3/καταχωρητές, κρύα token KV στο Eram, υποστήριξη σελιδοποίησης τύπου PagedAttention.
4. **Speculative Decoding**: draft μικρό μοντέλο + target μεγάλο μοντέλο στο Ecpu, εκμεταλλευόμενος την ευκολία παραλληλισμού του Ecpu.
5. **Reasoning άγκιστρα**: αλυσίδα reasoning (CoT / στοχασμός / tree-of-thought) μπορεί να εισαχθεί στον βρόχο συλλογισμού, χωρίς σύζευξη με τον πυρήνα της μηχανής.
6. **Αφαίρεση εντολών διαπλατφορμική**: ανίχνευση AMX / AVX-512 BF16/INT8 / AVX2 / NEON / SVE κατά την εκτέλεση, dispatch βέλτιστου πυρήνα.

## 4. Στρατηγική Σχεδιασμού

| Στρώμα | Ρόλος | Βασική σχεδίαση |
|---|---|---|
| **C πυρήνας** | Κύριο θεμέλιο απόδοσης | βιβλιοθήκη tensors, ποσοτικοποιημένοι πυρήνες, NUMA/threadpool, mmap φόρτωση βαρών, πύργος απόδοσης αμιγούς C |
| **Python προγραμματισμός** | Δευτερεύον στρώμα ενορχήστρωσης | PagedAttention, continuous batching scheduler, αφαίρεση συσκευής, block pool |
| **Αυξητικά μέρη** | Επιλεκτική ενίσχυση | RadixAttention (επαναχρήση προθέματος), αυξητικός detokenizer |

**Αρχή**: πρωτότυπος κώδικας, όχι fork, όχι αντιγραφή· επαναϋλοποίηση των βέλτιστων μοτίβων σχεδιασμού υπό την ενοποιημένη αφαίρεση Ecpu-Eram.

### 4.1 Γιατί C πυρήνας ως κύριο

- Ο πυρήνας του έργου είναι **CPU συλλογισμός**, και ο αμιγώς C πυρήνας είναι ο πύργος απόδοσης CPU
- Η βιβλιοθήκη tensors είναι κομψή: μηδενικό overhead αφαίρεσης, ρητή διάταξη μνήμης, dequant κατ' απαίτηση
- Οι GGUF ποσοτικοποιημένοι πυρήνες (Q4_K_M κλπ.) είναι η πιο ώριμη λύση χαμηλής ακρίβειας σε CPU
- NUMA συγγένεια + threadpool + mmap φόρτωση βαρών, ευθυγραμμίζεται άμεσα με τον στόχο βελτιστοποίησης εύρους ζώνης Eram
- Αμιγές C εύκολο να ενσωματωθεί, χωρίς C++ runtime εξάρτηση, βολικό για μικτή μεταγλώττιση

### 4.2 Γιατί Python στρώμα προγραμματισμού ως δευτερεύον

- Το στρώμα προγραμματισμού πρέπει να είναι ευέλικτο και ευέλικτο στην επανάληψη
- Ο **scheduler / block pool / continuous batching** σχεδιασμός είναι αμιγώς Python και ανεξάρτητος συσκευής, άμεσα μεταφέρσιμος
- Η αφαίρεση block του PagedAttention ταιριάζει για σελιδοποίηση KV στο Eram

### 4.3 Δέσμευση διεπαφής μικτού συλλογισμού GPU

Κατά τον σχεδιασμό της βάσης δεσμεύεται η αφαίρεση `Device` API, μελλοντικά το `CudaDevice` μπορεί να ενσωματωθεί:

```
Αφαίρεση στρώματος συσκευής (src/runtime/device/)
├── base.py          # Device ABC: alloc/free/memcpy/synchronize
├── ecpu_device.py   # Υλοποίηση Ecpu (τρέχον κύριο)
├── cuda_device.py   # Δέσμευση CUDA (μελλοντική υλοποίηση)
└── registry.py      # Καταχώριση και ανίχνευση συσκευών
```

Επίσης δεσμευμένο στην πλευρά C πυρήνα:
```c
// ecpu/device/api.h
typedef enum { EDEV_ECPU, EDEV_CUDA, EDEV_COUNT } edevice_t;
typedef struct edevice_vtable {
    void* (*alloc)(size_t bytes);
    void  (*free)(void* ptr);
    void  (*memcpy_h2d)(void *dst, const void *src, size_t n);
    int   (*synchronize)(void);
    // Μελλοντική υλοποίηση CUDA συμπληρώνει αυτά τα function pointers
} edevice_vtable;
```

Προς το παρόν υλοποιείται μόνο το `EDEV_ECPU`, το `EDEV_CUDA` αφήνει κέλυφος vtable, μελλοντικά συμπληρώνοντας cuMalloc/cuMemcpy για ενσωμάτωση.

## 6. Σύστημα Μεταγλώττισης (μικτή μεταγλώττιση)

- **Windows**: χρήση WinAPI (`windows.h`, `CreateFileMappingA`, `GlobalMemoryStatusEx`, `_aligned_malloc`)
- **Linux**: εγγενής μεταγλώττιση (`sysinfo.h`, `posix_memalign`, `mmap`)
- Η διακλάδωση γίνεται με `#if defined(_WIN32)` στο C στρώμα, χωρίς εξάρτηση τρίτου μέρους
- CMake + MinGW (Windows) ή GCC (Linux) δημιουργούν `libecpu.dll` / `libecpu.so`

## 7. Ανοιχτά Ερωτήματα

> Ερωτήματα προς επιβεβαίωση.

- [Q1] **Σειρά μοντέλων**: Qwen3-1.7B (BF16, γρήγορο pipeline) → Qwen3-8B-Coder GGUF (επικύρωση ποσοτικοποιημένου πυρήνα) → Qwen3.5-9B (μέτρηση εύρους ζώνης);
- [Q2] **Τρόπος δέσμευσης C ABI**: ctypes (μηδενική εξάρτηση) / CFFI (πιο γρήγορο) / pybind11 (πιο ευέλικτο αλλά βαρύ); Προτίμηση ctypes αρχικά.
- [Q3] **NUMA συγγένεια Eram**: υποστήριξη διαμοιρασμού βαρών πολλαπλών NUMA κόμβων σε multi-socket CPU, ή πρώτα βελτιστοποίηση εντός ενός NUMA τομέα;
- [Q4] **Εύρος Speculative**: draft και target ίδιου μοντέλου διαφορετικής ποσοτικοποίησης, ή διαφορετικών μεγεθών;
- [Q5] **Μορφή Reasoning άγκιστρου**: υλοποίηση μέσω stop-token + ένεση προτύπου (ελαφρύ), ή ενσωματωμένο reasoning-state μηχάνημα (βαρύ);
